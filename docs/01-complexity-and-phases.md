# 01-complexity-and-phases.md

# AI 项目接管训练器：复杂度评估、阶段拆分与验收标准

## 1. 文档定位

本项目不是商业团队用来验证市场的最小可用 Demo。

本项目是一个自用优先、能力提升优先、长期可维护的真实项目代码训练系统。

因此，本项目的阶段拆分不是为了砍功能，不是为了快速试错，而是为了让 Cursor Composer / AI 编程工具在有限上下文内稳定实现一个完整系统。

阶段的含义是：

> 工程实现顺序，而不是产品愿景缩水。

最终目标从一开始就保持完整：

- 接入已有 AI Gateway / AI 网关能力。
- 使用 Cloudflare 生态。
- 使用 React Router / Remix 新体系。
- 使用 D1 持久化。
- 支持完整课程、题目、AI 讲解、AI 出题、错题本、能力树、阶段考试。
- 覆盖前端状态链、后端请求链、前后端业务链。
- 最终服务于读懂并接管用户自己的真实项目。

---

## 2. 产品复杂度总评

如果按完整可用版本计算，本项目复杂度约为 8/10。

复杂的地方不在页面多，而在以下几点：

1. 题目不是普通题库，而是围绕真实项目代码因果链设计。
2. 题型不止选择题，还包括排序、填空、纠错、分支推演、位置判断。
3. 需要 AI 讲解，但 AI 不能失控，必须结构化、可审核、可复用。
4. 需要同时训练前端和后端逻辑。
5. 需要记录错题、能力树、阶段考试结果。
6. 后续可能导入真实项目代码，复杂度会进一步提升。

所以工程上必须分批实现，但文档上要完整描述。

## 4. 完整第一版的产品边界

完整第一版不是静态 Demo。

完整第一版应该包含：

- D1 数据库。
- 课程/关卡/题目正式数据模型。
- 用户进度。
- 错题本。
- 能力树。
- 阶段考试。
- AI Gateway 调用。
- AI 提示与讲解。
- AI 出题草稿。
- 管理员审核发布。
- 至少 3 个真实项目案例课程。
- 前端状态链训练。
- 后端请求链训练。
- 前后端连接训练。

完整第一版暂不包含：

- 自动扫描整个 GitHub 仓库。
- AST 深度分析。
- 在线 IDE。
- 多人社区。
- 商业化付费。
- 移动端原生 App。

这里“不包含”不是因为做 MVP，而是因为这些不是当前自用能力提升的必要核心。

## 5. 阶段拆分总览

为了适配 Cursor Composer 的执行能力，建议拆成 5 个工程阶段。

### Stage 0：文档与骨架阶段

目标：

- 建立文档。
- 建立类型。
- 建立目录。
- 建立数据模型。
- 不急着写完整 UI。

产物：

- docs/project-code-coach 下完整文档。
- app/lib/learn 类型。
- app/lib/server/learn 服务接口草案。
- migrations SQL 草案。
- AI Gateway Adapter 接口草案。

验收：

- 文档完整。
- 类型清晰。
- Cursor 能理解范围。
- 没有开始乱写业务。

---

### Stage 1：核心学习系统阶段

目标：

- 实现完整学习系统的基础体验。
- 使用 D1 作为正式数据来源。
- 支持本地 seed 课程数据。
- 支持用户答题、错题、能力树。

必须包含：

- 课程列表。
- 课程详情。
- 关卡练习。
- 多题型组件。
- 答案校验。
- 中文解释。
- D1 数据读写。
- 用户进度。
- 错题本。
- 能力树。

验收：

- 登录用户答题记录写入 D1。
- 未登录用户可以试用，但核心进度以登录用户为准。
- 每个题目都有 explanation。
- 错题能复习。
- 能力树会变化。

---

### Stage 2：AI 讲解与提示阶段

目标：

- 接入已有 AI Gateway / AI 网关。
- 用户答错后可以请求 AI 进一步讲解。
- 用户可以点击“给我一个提示”，但不是直接给答案。
- AI 讲解要围绕当前题目、当前错误答案、当前能力标签。

必须包含：

- AI Gateway Adapter。
- 提示生成接口。
- 讲解生成接口。
- 调用日志。
- 限流。
- 错误兜底。
- 前端 AI 提示面板。

验收：

- 用户答错题后可以获得 AI 中文讲解。
- AI 不改代码。
- AI 不越权访问其他题目。
- AI 调用失败时显示友好提示。
- AI 调用记录可追踪。

---

### Stage 3：AI 出题草稿与审核阶段

目标：

- 根据代码片段生成训练题草稿。
- 管理员审核后发布。
- 让用户能把自己项目里的真实代码逐渐转成课程。

必须包含：

- 粘贴代码片段。
- 选择训练目标。
- 选择题型。
- AI 生成结构化 JSON。
- JSON Schema 校验。
- 管理员编辑题目。
- 审核发布。
- 拒绝草稿。
- 保存来源文件路径和上下文说明。

验收：

- AI 生成的题目不会直接发布。
- 不符合 schema 的输出不能入库。
- 管理员可以修改后发布。
- 发布后普通学习页能看到题目。

---

### Stage 4：阶段考试与真实改造任务阶段

目标：

- 把学习结果转化为真实项目改造能力。
- 考试不只是答题，而是模拟真实需求拆解。

必须包含：

- 阶段考试页面。
- 任务背景。
- 文件范围提示。
- 子任务拆解。
- 代码填空。
- 请求链排序。
- 后端分支推演。
- 最终复盘。
- 通过/未通过判断。

验收：

- 至少 3 个考试：
  1. 全局主题系统改造。
  2. 登录保护 API 设计。
  3. 前后端完整功能设计。
- 用户能看到自己未掌握的能力点。
- 考试结果写入 D1。

---

### Stage 5：项目代码导入辅助阶段

目标：

- 不做完整自动仓库分析，但支持用户手动提交代码片段、文件路径、业务说明。
- 系统辅助生成训练素材。

必须包含：

- 代码片段库。
- 来源文件记录。
- 代码片段归档。
- 从片段生成题目。
- 关联到课程/关卡。

暂不做：

- 自动 GitHub 扫描。
- 自动 AST 全项目分析。
- 自动跨文件依赖追踪。

验收：

- 用户能把个人网站某个真实文件片段保存进系统。
- 用户能基于片段生成题目草稿。
- 用户能把题目发布到课程。

---

## 6. 核心验收标准

完整第一版完成后必须满足以下标准。

### 6.1 学习功能验收

- 用户能进入学习中心。
- 用户能看到课程、关卡、题目。
- 用户能完成至少 3 个完整课程。
- 支持至少 7 种题型：
  - 单选。
  - 多选。
  - 排序。
  - 填空。
  - 纠错。
  - 分支推演。
  - 位置判断。
- 每道题有中文解释。
- 每道题有能力标签。
- 每次答题有结果反馈。

### 6.2 数据功能验收

- 课程数据存 D1。
- 题目数据存 D1。
- 答题记录存 D1。
- 错题存 D1。
- 能力分数存 D1。
- 阶段考试结果存 D1。
- 支持 seed 初始课程。

### 6.3 AI 功能验收

- 接入已有 AI Gateway。
- 用户答错后可请求 AI 讲解。
- 用户可请求渐进提示。
- 管理员可用 AI 生成题目草稿。
- AI 输出必须结构化。
- AI 题目必须审核后发布。
- AI 调用有日志。
- AI 调用有限流。

### 6.4 项目训练验收

至少覆盖 3 个真实项目案例：

1. 黑夜模式与全局状态。
2. AI Chat 发送消息完整链路。
3. 登录保护与领取 token。

每个案例必须同时体现：

- 真实项目背景。
- 代码片段。
- 业务因果链。
- 常见 AI 改错方式。
- 正确架构解释。

---

## 7. 为什么不按商业 MVP 做

商业 MVP 追求：

- 最小功能。
- 快速验证市场。
- 快速失败。
- 快速迭代。
- 用户增长。

本项目追求：

- 自用有效。
- 能力提升。
- 长期可维护。
- 深度理解真实项目。
- 形成学习飞轮。

因此，本项目不以“能不能卖”为标准，而以“能不能让我读懂并接管自己的项目”为标准。

阶段拆分只是为了降低 AI 编程执行风险，不是为了砍掉完整功能。

---

## 8. 给 Cursor 的阶段策略

不要让 Cursor 一次性写全部代码。

但可以让 Cursor 一次性读完整文档。

推荐执行方式：

1. 先让 Cursor 创建类型、数据库 migration、服务接口。
2. 再实现学习页面与题型组件。
3. 再接 D1。
4. 再接 AI Gateway。
5. 再做 AI 出题和审核。
6. 再做阶段考试。
7. 最后做代码片段库。

注意：

> 文档要完整，执行要分批。
